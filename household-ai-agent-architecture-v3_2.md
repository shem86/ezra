# Household AI Agent: Architecture Plan v3

> **Purpose:** a self-contained context handoff. A fresh agent should be able to read this document alone, with no earlier drafts, and continue planning the project and then generate the project docs (requirements, PRD, ADRs). This is the single source of truth; ignore any prior versions. Each section below is a locked decision with its rationale and, where useful, the tempting alternative and why it was rejected. The load-bearing decisions cluster around five places where this class of system tends to fail quietly: the ingestion boundary (durably capturing an inbound message before acking it), the human boundary (approvals), the external-service boundary (idempotency), the recovery boundary (restore semantics), and the failure-detection boundary (monitoring). Read those closely.

## Project

An AI agent household assistant with two goals:

1. **Learning:** hands-on experience with a production agentic system: architecture, orchestration, tool integration, retrieval, model eval, session isolation, subagent delegation, self-improving feedback loops.
2. **Utility:** a real daily tool for the builder and his wife, used through a shared WhatsApp group.

**Governing principle:** learning and utility pull in opposite directions (sophistication vs reliability). Build a reliable core first; add the "exercise the muscle" pieces (subagents, self-improving loops) as deliberate later experiments, never in the v1 critical path.

**Blast-radius note (informs v1):** this is a shared tool a second person relies on, not a single-user toy. The ban/failure probability of the transport is the same as a personal deployment, but the consequence is not: a silent failure means she stops getting reminders and you hear it from her, not your logs. So v1 keeps the operational safety net (independent health monitoring, down-alerts, a fallback path) that you might skip for a personal experiment. Every v3 boundary fix traces back to this note: the failures that matter here are silent ones at the edges, not crashes in the core.

**Builder context:** senior distributed-systems architect, works in TypeScript, values reliability and determinism, heavy Claude Code user. No existing cloud infra for this project (no GKE).

## v1 capability scope (confirmed)

Reminders / scheduling, shared lists (groceries / todos), calendar read/write, and memory-backed household Q&A. Everything else is later-phase.

## Locked decisions

### 1. Transport: WhatsApp via Baileys (unofficial), dedicated burner number
- **Decided.** Baileys gives full group support and free proactive messaging; the official Cloud API's 24h window and template-only proactive messages would gut proactivity and self-scheduling.
- **Risk profile:** TOS/ban risk, lowest-risk shape available (private 2-person group, both saved contacts, high reply ratio, low volume). Residual risk is protocol fingerprinting plus Meta's general-purpose-AI-assistant restriction (enforcement aimed mostly at public official-API bots).
- **Mitigations:** treat the number as disposable; warm it 1 to 2 weeks (use it like a human before the bot takes over); human-like send delays; pin and lockfile every WhatsApp-adjacent npm package (the transitive tree is the real attack surface, not just the top-level dep).
- **Operational requirement:** socket-health monitoring with a down-alert. The alert path must be independent of the socket it monitors (see decision 7). A dropped socket is an incident, not a silent state.
- **Session state: high-value secret, persisted, re-pair on loss.** The Baileys auth state (pairing credentials plus the Signal-protocol ratchet keys) is a live "message anyone as you" credential, more sensitive than the calendar OAuth tokens. Three properties follow. It must persist across restarts so the number does not re-pair on every deploy (in a writable volume, since the root filesystem is read-only per decision 7). It is secret-class and encrypted at rest like other secrets. And its recovery action on loss or corruption is re-pair via QR, never restore from backup: decision 8 explains why a stale ratchet snapshot is restore-hostile.
- **Soak test, staged.** The connection surviving human traffic is the cheap assumption; the number surviving bot traffic patterns is the expensive one, and the human-to-bot fingerprint shift is the hazard you actually identified. A human-traffic soak cannot test it. Stage it:
  - **Stage A (human soak):** run Baileys on the burner under realistic 2-human traffic for the warming window. Confirms SIM, number, and connection survive human use.
  - **Stage B (low-rate bot soak):** introduce bot-pattern traffic at minimal rate (one proactive send per day, human-like send delays, at least one forced reconnect) for several days. This is the assumption worth the most.
  - **Stage C (ramp):** increase proactive cadence toward expected v1 volume, watching for enforcement.
  - Do not wire the agent in on top of an unsoaked transport. This is still the riskiest assumption in the project.
- **Cost:** about $20 one-time for a physical prepaid SIM (VoIP/virtual numbers get blocked at registration). Recurring cost is host plus Claude tokens.

### 2. Sessions and concurrency: single shared group, explicitly serialized
- There are **three writers to shared state**: each spouse, and the agent's own proactive jobs. "Sequential human turns" is not a given. Two people in a group send interleaved messages, WhatsApp arrival order is not intent order, and a single person fires rapid-bubble messages that are one thought.
- **Design:** one single-consumer durable workflow per conversation (a DBOS workflow queue keyed by conversation, concurrency 1). All inbound messages and all proactive jobs enqueue to it; exactly one item runs to completion at a time. This gives a total order on state mutations and removes lost-update races on lists/reminders.
- **v1 ordering:** FIFO, no preemption. If a proactive job is in flight when a human message arrives, the human waits the few seconds for it to finish. Interruption/cancellation is a large complexity jump and is out of scope for v1.
- **Critical interaction with HITL:** a workflow must never block the single slot on a human. Approvals are parked out-of-band and never occupy the consumer (decision 10). This is what keeps "human waits a few seconds for a proactive job" from degrading into "conversation frozen indefinitely behind an unanswered approval."
- **Ingestion durability (the fifth boundary).** Everything else in this decision covers what happens after a message is on the queue; the path from Baileys receiving a message to the durable enqueue is itself a boundary that can lose data silently. Durably enqueue the inbound message before acknowledging it to WhatsApp. The ack point is the lever: WhatsApp redelivers un-acked messages on reconnect, so an un-acked message survives a crash, while an acked-but-not-enqueued message is gone with no error and nobody knows until the groceries are missing. Do not ack until the message is durably enqueued.
- **Inbound debounce is a consumer-side property, not a pre-enqueue timer.** Batch a sender's rapid bubbles (a short silence window, about 1.5 to 3 seconds) into a single turn so the loop sees one thought, not five, which also kills the rapid-fire race. Critically, do this by how the single consumer pulls and groups messages that are already durably enqueued, not with an in-memory timer that holds messages before the enqueue. An in-memory pre-enqueue debounce reintroduces exactly the silent-loss window above: a crash mid-window evaporates "add milk" with no error. Enqueue durably first, debounce on the way out.
- **Self-echo filtering.** Baileys delivers the bot's own outbound messages back as inbound (the `fromMe` flag). Filter these at ingestion or the agent triggers on its own reminders. This is a day-one bug everyone hits; named here so it is not rediscovered.
- **Ordering note (recorded so it is not rediscovered):** per-sender debounce reorders batches relative to wall-clock arrival, and the total order is by enqueue time, not send time. This is fine for this use case, but the "total order" the queue guarantees is over enqueue order, not over the order messages physically arrived.

### 3. Orchestration / durable execution: DBOS
- TS-native; embeds as a library in the Node process (no separate orchestrator or cluster); checkpoints to Postgres; fits "no infra." Same durable-execution mental model as Temporal with far less ceremony. It trusts the developer (no enforced determinism sandbox), acceptable here only with the two determinism guardrails below.
- **Mental model:** durable execution replays the orchestration function from the top on recovery, skipping already-completed steps. Invariant: orchestration control flow must be deterministic, and every nondeterministic or side-effecting operation (I/O, randomness, clock, LLM calls) must live inside a recorded step.
- **What idempotency covers:** idempotency and exactly-once fall out of this **for steps that re-run internally**. They do not fall out for external effects that the outside world cannot dedupe. Those are handled explicitly in decision 4 and decision 10. Do not treat "DBOS gives exactly-once" as covering the WhatsApp send or the calendar write.
- **Determinism guardrails (the substitute for an enforced sandbox).** DBOS will silently accept a nondeterministic workflow body; the violation surfaces only when a recovery replay diverges, which is rare, in production, at the worst moment. Two cheap controls replace most of what Temporal's sandbox would give you: (1) a custom ESLint rule that bans clock reads, randomness, env-driven branches, and direct I/O inside any `@DBOS.workflow` function, failing CI on violation; (2) a recovery eval scenario (decision 9) that kills a workflow mid-flight, replays it, and diffs the output against the non-interrupted run. Without these, "acceptable here" rests on never making the mistake, which is not a control.

### 4. Reasoning layer and agent loop: Vercel AI SDK Core only, DBOS owns the loop
- Use Core at the primitive level (`generateText` / `generateObject` + Zod tool definitions) as the per-turn unit. Do **not** use Core's high-level `Agent` / `ToolLoopAgent` to run the whole multi-step loop; that hides the loop in one DBOS step and kills per-turn journaling and clean mid-loop human-in-the-loop.
- **The Claude Agent SDK (Anthropic's Claude Code harness) is also out, for the same reason plus a billing one.** It owns the agentic loop exactly as Core's high-level agent does, so it breaks the same per-turn journaling and fire-and-fold HITL. It is additionally the only path to the subscription's Agent SDK credit, which can tempt a switch on cost grounds; that temptation is analyzed and declined in decision 6. The reasoning layer stays on Core primitives with DBOS owning the loop regardless of billing.
- **No `@anthropic-ai/sdk` fallback, with one named gate.** Committed to the provider-agnostic abstraction. Bleeding-edge Anthropic-specific features reach the app through Core's provider-passthrough options, possibly with a lag, and that is accepted. **Exception to verify before locking:** prompt caching is load-bearing for cost (the loop re-sends growing context). Confirm Core actually threads `cache_control` breakpoints cleanly through provider passthrough. If it cannot express cache breakpoints, that single fact justifies the escape hatch, because the cost-control plan depends on it. Verify this during the spike, not after.
- **Loop shape (DBOS owns it):**
  ```ts
  @DBOS.workflow()
  async function handleTurn(convId: string, batch: InboundMsg[]) {
    let msgs = await DBOS.runStep(() => loadContext(convId));   // step
    msgs.push(...toModelMsgs(batch));
    for (let i = 0; i < MAX_ROUNDS; i++) {                      // hard cap
      const turn = await DBOS.runStep(() => callModel(msgs));   // step: records assistant msg + tool_use ids
      msgs.push(turn.assistant);
      if (turn.toolCalls.length === 0) break;
      let parked = false;
      for (const tc of turn.toolCalls) {
        const res = await runTool(tc, convId);                  // each tool exec = its own step; a confirm-before tool records a pending action and returns a synthetic "pending" result
        msgs.push(res);                                         // ALWAYS push a result, even on deny or park
        if (res.parked) parked = true;                          // park is known from the tool's risk tier inside runTool, not from the model output
      }
      if (parked) break;                                        // fire-and-fold: end the turn, free the slot (decision 10)
    }
    await DBOS.runStep(() => persistContext(convId, msgs));
  }
  ```
- **Invariants that keep this correct on replay:**
  - Variable iteration count is fine. The loop is deterministic on replay because each branch input (the model output) comes from a recorded step.
  - Nothing nondeterministic in the workflow body: no clock reads, randomness, env-driven branches, or retry jitter in orchestration. Those live inside steps.
  - The model-call step records the assistant message and its `tool_use` IDs atomically. On replay you must get the same IDs back or `tool_result` correlation breaks. Do not regenerate IDs or post-process model output impurely outside a step.
  - Every `tool_use` is answered with a `tool_result`, including denials and parks. On a confirm-before park, inject a synthetic "pending approval, action_id=X" result and end the turn (decision 10). On a deny, inject a synthetic "user declined" result. An unanswered `tool_use` makes the next model call error.
  - Tool-execution idempotency: a crash mid-step (pre-commit) re-runs the step on recovery, so internal tool side effects must dedupe. Derive an idempotency key from `(workflowID, stepNumber)`. This covers internal re-execution only; external effects need decision 10's delivery semantics on top.
  - Do not journal the whole transcript per step. `msgs` is workflow-local and rebuilds from each step's recorded output (a per-turn delta) on replay. Pass steps only what they need; never pass the full array as a step input.
  - Tools within a turn run serially unless provably independent (parallel steps touching shared state reintroduce the race from decision 2).
- **`MAX_ROUNDS` cap-hit behavior.** Hitting the cap must not break the loop silently. On cap-hit, force one final no-tools model turn to emit a user-facing message (a summary, or "I got stuck on X, want me to keep going?"). A silent stall at the cap is exactly the kind of silent failure the blast-radius note forbids.
- **Cost controls (the loop re-sends growing context, so address it):**
  - Prompt caching: mark the stable prefix (system + tool defs + early turns) with `cache_control` via Core's provider passthrough so loop iterations and subsequent turns hit the cache instead of re-billing the prefix. Gated by the verify step above.
  - Compaction: when the transcript crosses a threshold, run a deliberate summarize-and-truncate step that folds older turns into the semantic store. **This step writes to an external store, so it carries the same idempotency discipline as a tool:** key the semantic-store write on `(workflowID, stepNumber)` so a crash-replay mid-compaction does not double-write the summary. Recorded, so deterministic on replay.
  - `MAX_ROUNDS` hard cap on tool rounds so a confused model cannot run up cost.

### 5. Memory: tiered
- **Deterministic structured store** (Postgres tables): source of truth for anything that must be exact: lists, schedules, reminders, household facts and preferences. The LLM reads/writes through typed tools, never via semantic recall.
- **Semantic store** (pgvector): fuzzy episodic recall of past conversations. Lossy by nature, which is fine.
- **Datastore topology (see decision 8).** The durable-execution journal is separated from the source-of-truth state so they can be restored independently. The semantic store can co-reside with the structured store; the journal is the one that must be separable.

### 6. Model: Claude
- Haiku-class for routing and cheap classification (this includes the approval-relatedness classifier in decision 10); Sonnet/Opus-class for real reasoning.
- **Model access and billing: Console API key, separate from any subscription.** The production agent calls Claude through Vercel AI SDK Core, which uses the Anthropic API with an API key billed to the Console account, metered per token. This is a separate pool from any Claude Pro/Max subscription, which provides no API credits or discount for programmatic use. The cost levers are decision 4's prompt caching and compaction plus Haiku-class routing, not the billing channel.
- **The subscription's programmatic credit is deliberately not used for the runtime.** Subscriptions include a modest monthly credit, sized to the tier, for subscription-authenticated programmatic paths (the Claude Agent SDK and `claude -p` headless), but it is billed at full API rates with no per-token discount, so its only benefit is prepayment. Reaching it would require building the reasoning layer on the Agent SDK harness, which owns the loop and therefore reverses decision 4 (loop hidden in one step, per-turn journaling and fire-and-fold HITL lost), adds Claude-only lock-in, and forces a fragile headless subscription-OAuth token on the always-on box instead of a stable API key, with its own ToS questions for unattended 24/7 use. That trade is not worth a small prepaid credit for a reliability-first shared tool. The credit instead gets consumed naturally by headless dev automation while building this (`claude -p` scripts, CI, non-interactive skills).

### 7. Host and failure detection: cheapest always-on, no paid PaaS
- **Always-on is mandatory** because the WhatsApp socket must stay connected. A dropped socket causes reconnect churn (a mild ban signal) and missed messages.
- **Oracle Always Free + Pay-As-You-Go conversion is a candidate $0 path,** but the claim that PAYG conversion exempts the account from 7-day idle reclamation is a **vendor-policy assertion to re-verify at provisioning time, not a locked fact**. The host decision is the one place a stale fact fails silently and cheaply-looking: you will not notice the policy changed until the box is reclaimed. Confirm against Oracle's current docs the week you provision.
- **Residual Oracle risk** (separate from reclamation): "out of host capacity" on A1 ARM shapes can block provisioning in your region, and Oracle is opaque about account terminations (mostly signup/identity/TOS driven).
- **Hetzner (about EUR 4/mo)** is the named fallback and the choice if you want to skip Oracle's provisioning friction and policy ambiguity. At two-user scale the cost is noise against the reliability of the foundation everything else sits on.
- **Decision:** Oracle PAYG if you can provision A1 and the current reclamation policy still holds; otherwise Hetzner.
- **Failure detection must not share fate with what it monitors (load-bearing).** Two independent requirements:
  - **Independent alert transport:** the down-alert for a dropped WhatsApp socket must not ride the same Baileys socket. Use a separate channel (Telegram bot, Pushover, SMS, or email). If the alert path is the socket, it is silent exactly when you need it.
  - **External dead-man's-switch:** an in-process health monitor cannot report that its own process or host died. Add an external check (a healthchecks.io-style ping the box must emit on a schedule, alerting from outside when the ping stops). Host death is then detected externally, not by a monitor that dies with the host.
- **Process hardening and egress allowlist (contains the supply-chain blast radius).** The agent runs in one Node process holding Postgres creds, the Anthropic key, Google OAuth refresh tokens, and a live WhatsApp session that can message anyone as you. A compromised transitive dep in the Baileys tree (deferred risk 3) inherits all of it. A syscall-level sandbox (gVisor, microVM) is overkill at two users; standard hardening is the right weight: run the process as a non-root user, a read-only root filesystem with explicit writable volumes only for the Baileys session state (decision 1) and the co-located journal Postgres (decision 8), no added capabilities, secrets injected at runtime rather than baked into the image. The highest-leverage single control is an **egress allowlist**: a malicious dep almost always needs outbound network to be useful (exfiltrate the OAuth tokens, phone home), so restrict outbound to the endpoints the agent actually talks to (Anthropic, Google, the Neon host for the state store, the alert channel, and WhatsApp) and most of the risk evaporates for near-zero cost. One caveat: WhatsApp's endpoint surface is wide (signaling plus media CDNs) and not a tidy hostname list, so that entry needs some iteration to pin down (see burner hygiene in open items); it is still worth doing. This pairs with the secret-class encryption in the deferred risks: hardening limits what a compromise reaches, encryption limits what it reads.

### 8. Data durability and recovery
- The structured store is the source of truth. v3's topology separates it from the DBOS journal.
- **Why separate:** the tempting setup is to put the journal, structured state, and pgvector in one Postgres and rely on managed PITR to cover durability. That does not retire the blast radius. PITR restores journal and state together to one point, and external side effects already emitted (WhatsApp messages sent, calendar events written) do not roll back with the database. Restoring state to T-1h while the journal rolls back with it can re-drive in-flight workflows against a world that already moved, re-emitting effects. Durable execution assumes the world only moves forward; a state restore violates that assumption.
- **Topology:** the DBOS journal lives in its own Postgres co-located on the box; the source-of-truth structured state and the pgvector semantic store live in managed Postgres. Different databases, so a state-corruption restore never forces a journal rollback and vice versa. This split also falls out of a cost and an availability reality, not only the restore argument, both below.
- **Backups:**
  - **State and semantic stores: managed Postgres + pgvector (Neon), with PITR.** Automated backups and point-in-time recovery built in, which deletes the hardest reliability chore. These stores are touched only on message turns, so they can autosuspend and may fit a free or cheap tier. Cost: low-tens-of-ms latency (negligible at two users) and data on a third party (relevant to secrets at rest, see deferred risks).
  - **Journal: co-located Postgres on the box, not managed.** The journal is rebuildable operational state, not the source of truth: losing it loses only in-flight workflows, which is recoverable churn, not household data. It needs neither PITR nor a third party. If you want belt-and-suspenders, ship periodic encrypted dumps to S3-compatible storage (Backblaze B2 or Cloudflare R2), but it is not the priority the state store is.
  - **Why the journal is not on Neon (cost):** DBOS polls the journal continuously (queue polling, scheduled-workflow timers), so a managed journal compute never autosuspends. Always-on at minimum compute is on the order of ~180 CU-hours/month, which exceeds Neon's free allowance (~100 CU-hours) before the state store uses anything. The tempting "two Neon databases, free at this scale" plan is very likely not free. Treat the exact numbers as a **verify-at-provisioning flag, not a locked fact**, the same caution decision 7 gives Oracle's reclamation policy.
  - **Why the journal is not remote (availability):** the agent cannot make progress if no step can be recorded, so a remote journal couples the agent's liveness to Neon's reachability and a network blip halts everything. Co-locating the journal removes that coupling. The state store being remote is fine because it is read or written only per turn, not polled continuously.
- **Recovery runbook (required):** any restore must reconcile external effects that did not roll back. The reconciliation tools are the deterministic external IDs and the sent-log from decision 10 (idempotent calendar event IDs, the WhatsApp sent-log keyed on action/idempotency key) plus revalidation-at-execution for order-sensitive actions. Write this runbook before you need it.
- **Baileys session state is restore-hostile (a second store the move-forward-only insight applies to).** The Baileys auth state (pairing creds plus Signal-protocol ratchet keys) is durable state, but the ratchet only advances: restoring a stale snapshot desyncs encryption and produces exactly the anomalous-client behavior the ban-avoidance work tries to prevent. So its recovery action is re-pair via QR, never restore from backup. Persist it only to survive a clean restart; never roll it back to an old point. This is the same insight as the journal-vs-state restore above, applied to a store that lives outside Postgres (decision 1 holds the secret-class and persistence details).
- **Rule either way:** a backup on the same disk is not a backup, and an untested backup is a hypothesis. Restore into a scratch DB and diff at least once.
- **Recommendation:** Neon (PITR) for the state and semantic stores; the DBOS journal on a co-located Postgres on the box; documented recovery runbook. One split that serves the restore argument, the cost reality, and the availability coupling at once.

### 9. Observability and eval
- Structured tracing from day one (Langfuse, clean TS SDK) so the eval and feedback loop has data.
- A small regression suite of canned scenarios. LLM-as-judge for response quality later.
- **New eval target:** the HITL park/resume path and the approval-relatedness classifier (decision 10) are both behaviors that need scenario coverage, because they are where the model now experiences conversational discontinuity. Cover: approve-after-delay, deny, abandon-by-unrelated-message, refine-the-pending-action ("make it 4pm"), and stale-action-at-execution.
- **Recovery replay scenario:** kill a workflow mid-flight and replay it, asserting the replayed output matches the non-interrupted run and that no external effect double-fires. This is the runtime half of the determinism guardrails in decision 3; the lint rule catches violations statically, this catches them behaviorally.
- Self-improving feedback loops are deferred until eval exists.

### 10. Human-in-the-loop: out-of-band park, never block the slot
Read decision 2 alongside it; this decision is what keeps the serialized queue from deadlocking on a human.

- **Risk tiers unchanged.** Classify each tool: autonomous / notify-after / confirm-before, based on reversibility, cost, and third-party side effects.
- **Confirm-before never occupies the single consumer.** The tempting design is to block the workflow on `DBOS.recv()` until the human replies, but that holds the concurrency-1 slot and freezes the whole conversation and every proactive job behind an approval the human may never answer. Fire-and-fold avoids it.
- **Fire-and-fold mechanics:**
  1. The loop hits a confirm-before tool. It does **not** execute and does **not** block.
  2. It answers the `tool_use` immediately with a synthetic "pending, action_id=X" `tool_result` (preserving the never-leave-a-tool_use-unanswered invariant).
  3. It records the pending action out-of-band: a `pending_actions` row (action_id, conversation_id, serialized tool call, status, created_at, expires_at).
  4. It sends the approval prompt as the turn's closing message, then returns. The slot is now free; the queue keeps draining (reminders fire, the other spouse's messages process, the sender can keep chatting).
  5. Approval arrives as a new inbound event. A fresh turn looks up the action by action_id and executes via a transactional status transition: check-and-set `pending -> executing` in the same step that runs the tool, so a second approval (either spouse can approve, so two can arrive back-to-back) reads a non-pending row and no-ops. The concurrency-1 queue serializes the turns, but this status guard is what actually makes execution exactly-once. Set `executing -> done` after the tool runs. The real result is then fed into the model as new context (see the transcript note below). On deny, set `pending -> denied` and continue.
- **Transcript note (a natural wrong implementation to avoid):** the original `tool_use` was already answered with the synthetic "pending" `tool_result` in the parking turn, and that answer is final. The real result from the fresh turn therefore cannot be a second `tool_result` for that same `tool_use` id; answering one `tool_use` twice errors at the API. It enters the fresh turn as a new user- or system-role context message ("action X is now approved and done: <result>"). That is what "feeds the real result into the model as new context" means above.
- **Approval binding unchanged:** each pending action carries a short stable action ID; the prompt is sent as its own message; the user confirms via quoted-reply, which binds the confirmation to the right action even when more than one is outstanding. The action_id is the rendezvous key in `pending_actions`, which also removes the action_id-to-workflow_id tracking the inline `recv` design needed.
- **Unblocking is event-driven, not wall-clock.** Time and events answer different questions. Events answer "when do I stop holding the slot": the answer is immediately, on park. A new unrelated message is a **demotion** signal, not a **resolution** signal; in the fire-and-fold design the slot is already freed on park, so this collapses to "always park." Time answers only "when do I give up on an action nobody answered," which events cannot answer because the dominant failure (silence) produces no event.
- **Time as a loose soft backstop only.** `expires_at` exists solely to garbage-collect abandoned approvals. Because events do the latency-critical unblocking, the TTL can be generous (hours to a day) and soft: on expiry the action becomes non-offerable and the agent can surface it gently ("that booking I asked about earlier expired, still want it?"). Its exact value barely matters. This TTL is also what guarantees an unanswered approval eventually resolves rather than lingering forever.
- **External delivery semantics on execute (ties to decision 4 scope correction):**
  - **Calendar writes:** use a deterministic event ID derived from action_id (or `(workflowID, stepNumber)`). The create is then idempotent and safe to retry: effectively exactly-once.
  - **WhatsApp sends:** Meta offers no server-side idempotency, so pick a class per message. **Reminders, proactive nags, and the approval request prompt: at-least-once** (send-then-log against the action key; accept a rare duplicate on crash-between). A missed reminder is the exact blast-radius failure this project defends against, and a dropped approval prompt makes confirm-before fail silently (the action sits unanswered until TTL expiry), so for both a duplicate is the acceptable failure and a miss is not. **Post-hoc confirmation echoes and low-stakes replies: at-most-once is acceptable** (log-then-send). Keep a sent-log keyed on the action/idempotency key; the recovery runbook (decision 8) reads it.
- **Two new obligations this design creates (record so they are not rediscovered):**
  - **Relatedness classifier.** When a message arrives while an action is pending, classify: a refinement ("make it 4pm") should update the pending action, an unrelated message should not touch it, neither should silently auto-deny it. Haiku-class handles this (decision 6); it has its own error modes the inline timer did not have, hence the eval coverage in decision 9.
  - **Revalidation at execution.** Because the approve window can be long, re-check preconditions at execute time, not just at propose time, for order-sensitive actions (is 3pm still free, did the other spouse already add it). This is also a recovery-runbook tool (decision 8).
- **Rejected alternatives (with reasons):**
  - **Inline `recv` with a short hard timeout** (the cheapest build): keeps the turn synchronous and is genuinely fine *if approvals are answered in the same breath*. Rejected as the v1 default because this is a shared tool and the realistic case is one spouse asleep or at work when the agent proposes something, which is exactly when the inline design freezes the conversation. Kept on record as the fallback if instrumentation shows approvals are always near-instant.
  - **Resume-the-loop continuation** (store enough to re-enter `handleTurn` at round N): rejected outright. It manually reimplements the durable wait DBOS already provides, across workflow boundaries, and fights the framework. Fire-and-fold executes the tool in a fresh turn and feeds the result forward instead.
- **v1 policy default:** either spouse can approve.

### 11. Subagents and self-improving loops
- Learning-driven, deferred to later phases. Not in the v1 critical path. Add one specialized subagent later to exercise delegation and context isolation.
- **Code-execution isolation (conditional hard requirement).** v1 needs no code-execution sandbox because the tool surface is typed structured functions, not model-generated code to run. The moment a later phase adds any "run this" capability (a code interpreter, shell access, an arbitrary-script tool), that flips to a hard requirement and the execution must be isolated properly (separate sandboxed process or container, no access to the secrets or session the main process holds). Do not let a run-code tool inherit the main process's ambient authority.

## Stack summary (one line)

WhatsApp (Baileys, burner, staged-soak-tested, socket-health-monitored over an independent alert channel + external dead-man's-switch, session state persisted and secret-class with re-pair-not-restore recovery) -> Node/TS on an always-on cheap host (Oracle PAYG if reclamation policy verified, else Hetzner), the Node process hardened and egress-allowlisted -> DBOS durable execution as the spine, one single-consumer workflow per conversation (durable-enqueue-before-ack ingestion, serialized, consumer-side debounced, never blocked on a human) -> per-turn reasoning via Vercel AI SDK Core (Claude Haiku + Sonnet, called through a metered Console API key separate from any subscription) with DBOS owning the loop (model call and each tool exec as recorded steps; prompt caching gated on Core passthrough; compaction idempotency-keyed; MAX_ROUNDS with a forced final message) -> memory split: structured source of truth + pgvector semantic on managed Postgres (Neon, PITR), DBOS journal on a co-located Postgres on the box (rebuildable, not the source of truth) -> HITL via out-of-band fire-and-fold park (quoted-reply approval, event-driven unblocking, soft TTL backstop, per-effect delivery semantics, transactional execute-once guard, revalidation at execution) -> tracing via Langfuse, eval covering the park/resume and relatedness-classifier paths.

## Known deferred risks (accepted for v1, recorded so they are not rediscovered later)

- **Secrets in context, traces, and at rest:** household secrets stored as facts (wifi password and similar) can enter prompt context and Langfuse traces; OAuth refresh tokens for calendar sit in Postgres backups on a third party (Neon); and the Baileys session state (decision 1, the most sensitive of these) sits in a writable volume on the box. These are one concern (secret exposure) and should be unified: give the structured store a secret class that never enters prompts or tracing, and encrypt secret-class data at rest so neither a third-party DB and its backups nor a compromised box exposes it.
- **Privacy surface:** conversations flow through Baileys (decrypted on the box), Anthropic, Langfuse, and the Postgres host. Accepted for now.
- **Baileys supply chain:** the transitive dependency tree is the real surface. Pin and lockfile rather than only "vetting" the top-level package. Blast radius is now contained by the process hardening and egress allowlist in decision 7; pinning lowers the chance of compromise, hardening and egress restriction lower what a compromise can do.
- **Number handoff discontinuity:** the human-to-bot behavioral shift on the same number is itself a fingerprint signal; warming and the staged soak reduce but do not remove it.
- **Relatedness-classifier error modes:** the decision-10 classifier can misjudge refine-vs-unrelated. Accepted for v1; eval coverage (decision 9) is the guardrail.
